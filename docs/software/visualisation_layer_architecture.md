# H59 Grafana Dashboard Architecture Recommendation

## 1. Purpose

This document proposes a simple, local-first Grafana dashboard architecture for the existing h59 project.

The key design goal is to keep the **core H59 CLI tool clean and focused** while adding dashboard and Grafana support as a **separate optional component**.

The dashboard layer should:

- Reuse the SQLite database generated by the existing `h59` CLI.
- Avoid adding Grafana, Docker, or visualization dependencies to the core Python package.
- Keep dashboard assets version-controlled and reproducible.
- Follow time-series dashboard best practices.
- Stay simple enough for an agent to maintain.

## 2. Current Project Context

The existing project is already structured around a local-first CLI:

```text
h59
├── src/h59_client/       # installable CLI package
├── tests/                # unit tests
├── tools/                # small utilities
├── docs/software/        # user-facing software documentation
├── docs/research/        # reverse-engineering notes
├── data/                 # runtime database/output, not tracked
├── AGENT.md
├── README.md
├── pyproject.toml
└── uv.lock
```

The CLI already provides the right foundation:

- Device discovery and registration.
- One-shot and incremental sync into SQLite.
- Periodic sync daemon mode.
- Decoded measurements and raw protocol packet storage.
- Markdown reports and data completeness validation.
- Default database locations:
  - source checkout: `./data/h59.sqlite`
  - installed CLI: `$XDG_DATA_HOME/h59/h59.sqlite`
  - fallback: `~/.local/share/h59/h59.sqlite`

The dashboard architecture should treat this SQLite database as the primary interface between the core CLI and visualization layer.

## 3. Architectural Principle

The core CLI must remain independent from the dashboard stack.

```text
Core CLI responsibility:
  BLE sync
  device registry
  raw and decoded SQLite storage
  basic reports
  data validation

Dashboard responsibility:
  read-only visualization
  dashboard-ready SQL/views
  Grafana provisioning
  optional Docker Compose runtime
```

The Grafana component should never become required for:

- installing the CLI,
- syncing the device,
- running tests,
- generating CLI reports,
- maintaining protocol code.

## 4. Recommended Repository Layout

Add a clearly separated dashboard area under the repository root:

```text
h59
├── src/h59_client/
├── tests/
├── tools/
├── docs/
│   ├── software/
│   └── research/
├── data/
│   └── h59.sqlite                 # runtime only, not tracked
├── dashboard/
│   ├── README.md
│   ├── docker-compose.yml
│   ├── .env.example
│   ├── provisioning/
│   │   ├── datasources/
│   │   │   └── sqlite.yml
│   │   └── dashboards/
│   │       └── dashboards.yml
│   ├── dashboards/
│   │   ├── h59_overview.json
│   │   ├── h59_sleep_recovery.json
│   │   └── h59_data_quality.json
│   └── sql/
│       ├── views.sql
│       ├── dashboard_summary.sql
│       └── example_queries.sql
├── README.md
├── AGENT.md
├── pyproject.toml
└── uv.lock
```

### Why `dashboard/` at repository root?

This keeps visualization operational assets separate from the Python package.

Avoid putting Grafana files inside:

```text
src/h59_client/
```

because Grafana provisioning, Docker Compose, and dashboard JSON are deployment assets, not CLI library code.

## 5. Data Flow

Recommended first architecture:

```text
H59 bracelet
    ↓ BLE
h59 CLI / daemon
    ↓ writes
SQLite database: data/h59.sqlite
    ↓ read-only mount
Grafana SQLite datasource
    ↓
Local Grafana dashboards
```

The CLI writes data. Grafana reads data.

There should be no write path from Grafana into the H59 database.

## 6. Database Boundary

The SQLite database is the contract between the CLI and the dashboard.

### Core CLI owns

- schema creation,
- migrations,
- device registry,
- raw packets,
- decoded measurements,
- sync status,
- data completeness records,
- report-generation primitives.

### Dashboard owns

- read-only SQL views,
- Grafana panel queries,
- dashboard JSON,
- display conventions,
- metric labelling,
- visualization-specific aggregation.

If the dashboard needs additional views, prefer either:

1. SQL views defined in `dashboard/sql/views.sql`, or
2. future CLI-managed views if they prove broadly useful.

Do not add visualization-specific tables to the core schema prematurely.

## 7. SQLite First

For this project, SQLite is the right first dashboard-serving database.

Reasons:

- one user,
- one wearable,
- low data volume,
- local-first requirement,
- already used by the CLI,
- easy backup,
- easy inspection,
- simple deployment.

Do not introduce PostgreSQL, TimescaleDB, InfluxDB, or Prometheus in the first dashboard iteration.

A future migration to TimescaleDB or PostgreSQL can be considered only if SQLite becomes limiting.

## 8. Grafana Deployment

Grafana should be an optional service under `dashboard/`.

Example:

```yaml
# dashboard/docker-compose.yml
services:
  grafana:
    image: grafana/grafana-oss:latest
    container_name: h59-grafana
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      - GF_INSTALL_PLUGINS=frser-sqlite-datasource
      - GF_SECURITY_ADMIN_USER=${GRAFANA_ADMIN_USER:-admin}
      - GF_SECURITY_ADMIN_PASSWORD=${GRAFANA_ADMIN_PASSWORD:-admin}
      - GF_USERS_ALLOW_SIGN_UP=false
      - GF_ANALYTICS_REPORTING_ENABLED=false
      - GF_ANALYTICS_CHECK_FOR_UPDATES=false
    volumes:
      - ./grafana-data:/var/lib/grafana
      - ./provisioning:/etc/grafana/provisioning:ro
      - ./dashboards:/var/lib/grafana/dashboards:ro
      - ../data:/h59-data:ro
```

The database should be mounted read-only:

```text
../data:/h59-data:ro
```

Grafana should connect to:

```text
/h59-data/h59.sqlite
```

## 9. Grafana Datasource Provisioning

Create:

```text
dashboard/provisioning/datasources/sqlite.yml
```

Example:

```yaml
apiVersion: 1

datasources:
  - name: H59 SQLite
    type: frser-sqlite-datasource
    access: proxy
    isDefault: true
    jsonData:
      path: /h59-data/h59.sqlite
```

This keeps the datasource reproducible and avoids manual Grafana setup.

## 10. Dashboard Provisioning

Create:

```text
dashboard/provisioning/dashboards/dashboards.yml
```

Example:

```yaml
apiVersion: 1

providers:
  - name: H59 Dashboards
    orgId: 1
    folder: H59
    type: file
    disableDeletion: false
    editable: true
    updateIntervalSeconds: 30
    options:
      path: /var/lib/grafana/dashboards
```

Dashboard JSON files should live in:

```text
dashboard/dashboards/
```

## 11. Suggested Dashboards

Start with three dashboards only.

### 11.1 H59 Overview

Purpose: reproduce the main QWatch Pro-style cards.

Panels:

- Steps today.
- Latest heart rate.
- Heart rate over selected time range.
- Sleep duration.
- Sleep start/end.
- SpO₂ latest/range.
- HRV latest/trend.
- Stress score.
- Blood pressure estimate.
- Battery level.
- Last successful sync.

### 11.2 Sleep & Recovery

Purpose: show night-focused and trend-focused information.

Panels:

- Sleep duration over last 30 days.
- Sleep start and wake time.
- Night heart-rate average.
- HRV trend.
- SpO₂ minimum by night.
- Sleep timeline if stage/segment data is available.
- Deviation from 7-day or 30-day baseline, if available.

### 11.3 Data Quality

Purpose: prevent misleading interpretation from stale or incomplete data.

Panels:

- Last successful sync.
- Sync status history.
- Records inserted per sync.
- Samples per metric per day.
- Battery over time.
- Missing days.
- Metric freshness.
- Data completeness score if available from the CLI/reporting layer.

This dashboard should be treated as mandatory for the MVP.

## 12. Time-Series Query Best Practices

### 12.1 Keep Grafana Queries Simple

Grafana panel queries should read from clean tables or views.

Good:

```sql
SELECT
  timestamp AS time,
  value AS heart_rate_bpm
FROM sensor_samples
WHERE metric = 'heart_rate_bpm'
  AND timestamp BETWEEN $__from AND $__to
ORDER BY timestamp;
```

Bad:

```sql
SELECT complex_json_extraction_from_raw_packet_blob(...)
```

Raw packet decoding belongs in the CLI, not in Grafana.

### 12.2 Preserve Raw Data, Query Decoded Data

The dashboard should not query raw BLE payloads except in a debugging dashboard.

Preferred layers:

```text
raw_device_records
    ↓
decoded measurements / normalized samples
    ↓
summary views
    ↓
Grafana panels
```

### 12.3 Use Stable Timestamp Conventions

Prefer timestamp columns that are:

- ISO-8601 text or Unix epoch, consistently used,
- UTC internally where possible,
- convertible by Grafana into time-series panels.

If the core database already has timestamp conventions, the dashboard must adapt to them rather than inventing a second convention.

### 12.4 Precompute Heavy Aggregations Later

For the first version, live SQL over SQLite is fine.

If queries become slow, add summary views or materialized summary tables generated by the CLI/report command later.

Do not prematurely introduce a second database.

## 13. Dashboard SQL Layer

The dashboard may include optional SQL helper views in:

```text
dashboard/sql/views.sql
```

Examples:

```sql
CREATE VIEW IF NOT EXISTS dashboard_latest_metric AS
SELECT
  metric,
  value,
  unit,
  MAX(timestamp) AS timestamp
FROM sensor_samples
GROUP BY metric;
```

```sql
CREATE VIEW IF NOT EXISTS dashboard_daily_steps AS
SELECT
  date(timestamp) AS date,
  SUM(value) AS steps_total
FROM sensor_samples
WHERE metric IN ('steps_increment', 'steps')
GROUP BY date(timestamp);
```

Important: these are examples. The final view names and columns should match the actual current schema in `h59.sqlite`.

## 14. Integration with the Core CLI

The CLI should expose enough commands for dashboard operation without depending on Grafana.

Existing commands already cover most needs:

```bash
h59 sync -i
h59 sync -di --period 15m
h59 report 1 --output report.md
h59 device list
h59 daemon status
```

Optional future CLI command:

```bash
h59 db path
```

Purpose: print the resolved database path so dashboard scripts can locate it reliably.

Optional future CLI command:

```bash
h59 dashboard export-sqlite --output ./dashboard/data/h59.sqlite
```

Purpose: copy or snapshot the active database for dashboard use.

This is optional and should not be added until needed.

## 15. Operating Modes

### 15.1 Development Mode

Run sync manually:

```bash
h59 sync -i
```

Run Grafana:

```bash
cd dashboard
cp .env.example .env
docker compose up -d
```

Open:

```text
http://localhost:3000
```

### 15.2 Local Server Mode

Run the CLI daemon:

```bash
h59 sync -di --period 15m
```

Run Grafana as a separate Docker Compose service:

```bash
cd dashboard
docker compose up -d
```

The dashboard reads the same SQLite file in read-only mode.

## 16. Control Commands

Add a small `dashboard/Makefile` for convenience.

```makefile
up:
	docker compose up -d

down:
	docker compose down

logs:
	docker compose logs -f

restart:
	docker compose restart

open:
	open http://localhost:3000
```

Do not place these commands in the root Makefile unless the project already uses one.

Keeping the dashboard control surface inside `dashboard/` reinforces the separation from the CLI.

## 17. Privacy Defaults

Grafana should be configured for local use.

Recommended defaults:

```yaml
GF_ANALYTICS_REPORTING_ENABLED=false
GF_ANALYTICS_CHECK_FOR_UPDATES=false
GF_USERS_ALLOW_SIGN_UP=false
```

Deployment guidance:

- Do not expose Grafana to the public internet.
- Use LAN-only access or a private VPN.
- Change the default admin password.
- Mount the H59 database read-only.

## 18. MVP Acceptance Criteria

The dashboard MVP is complete when:

- `dashboard/docker-compose.yml` starts Grafana locally.
- SQLite datasource plugin is installed automatically.
- Datasource is provisioned from file.
- At least one dashboard is provisioned from JSON.
- Grafana reads the existing `data/h59.sqlite` file read-only.
- Overview dashboard shows the main H59 metrics available in the database.
- Data Quality dashboard shows last sync / freshness / sample coverage.
- No dashboard dependency is added to the core Python package.
- The CLI can be installed and used without Grafana.

## 19. Non-Goals for the First Version

Do not implement these yet:

- Custom React frontend.
- TimescaleDB.
- InfluxDB.
- Prometheus.
- Home Assistant integration.
- Grafana alerting.
- Multi-user support.
- Public internet deployment.
- Medical-grade interpretation.

## 20. Recommended Build Order

1. Inspect the current SQLite schema.
2. Identify existing decoded measurement tables.
3. Add `dashboard/` directory.
4. Add Docker Compose for Grafana.
5. Add SQLite datasource provisioning.
6. Add a minimal Overview dashboard using real table names.
7. Add Data Quality dashboard.
8. Add Sleep & Recovery dashboard.
9. Add optional SQL helper views if queries become repetitive.
10. Document how to run the dashboard from `dashboard/README.md`.

## 21. Guiding Rule

Keep the project split cleanly:

```text
h59 CLI = collect, decode, store, validate
Grafana dashboard = read, visualize, monitor
```

The dashboard should be an optional reader of the local database, not a new core subsystem.
